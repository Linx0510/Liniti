## 4 Разработка веб-приложения Liniti

### 4.1 Реализация веб-приложения

При реализации сайта **Liniti** в репозитории использован классический подход SSR (server-side rendering): backend на **Node.js + Express**, шаблоны на **EJS**, данные — в **PostgreSQL** через модуль `pg`.

Такое решение выбрано по двум причинам:
1. быстрое формирование HTML на сервере для сложных страниц (лента, профиль, карточка работы);
2. простая интеграция пользовательских данных (сессия, роли, likes, подписки) прямо в шаблон без отдельного frontend-фреймворка.

---

#### 4.1.1 Инициализация приложения и поток запроса

Входной файл `server.js` создаёт экземпляр Express-приложения, подключает middleware и маршруты. Логика обработки запроса строится в следующем порядке:

1. Парсинг тела запроса (`express.urlencoded`, `express.json`).
2. Раздача статики из `public/`.
3. Подключение `express-session` для хранения состояния пользователя.
4. Вставка текущего пользователя в контекст (`attachCurrentUser`).
5. Генерация/проверка CSRF-токена (`ensureCsrfToken`).
6. Передача запроса в роутеры (`pageRoutes`, `authRoutes`, `apiRoutes`, `adminRoutes`).
7. Обработчик 404, если маршрут не найден.

**Листинг 1 – Базовая конфигурация Express-приложения**

```js
app.set('view engine', 'ejs');
app.set('views', path.join(__dirname, 'views'));

app.use(express.urlencoded({ extended: true, limit: '50mb' }));
app.use(express.json({ limit: '50mb' }));
app.use(express.static(path.join(__dirname, 'public')));

app.use(
  session({
    secret: process.env.SESSION_SECRET,
    resave: false,
    saveUninitialized: false,
    cookie: {
      maxAge: 1000 * 60 * 60 * 24,
      httpOnly: true,
      sameSite: 'lax',
      secure: process.env.NODE_ENV === 'production',
    },
  })
);

app.use(attachCurrentUser);
app.use(ensureCsrfToken);

app.use(pageRoutes);
app.use(authRoutes);
app.use(apiRoutes);
app.use('/admin', adminRoutes);
```

В результате каждый HTML-ответ формируется уже с учётом контекста пользователя: авторизован ли он, какие кнопки показать (вход/профиль), какие действия доступны (лайк, жалоба, публикация работ и т. д.).

---

#### 4.1.2 Маршрутизация и разделение ответственности

Маршруты вынесены в отдельные файлы (`routes/*`), а бизнес-логика — в контроллеры (`controllers/*`). Такое разделение делает код сопровождаемым:

- **routes** отвечают за URL, middleware и переход к обработчику;
- **controllers** отвечают за SQL-запросы, валидацию и подготовку данных для шаблона;
- **views** отвечают за визуальное представление и частично за фронтенд-интерактивность.

**Листинг 2 – Примеры маршрутов пользовательской части**

```js
router.get('/', pageController.getIndexPage);
router.get('/lenta', pageController.getLentaPage);
router.get('/profile', pageController.getProfilePage);
router.get('/profile/:id', pageController.getProfilePage);

router.get('/works/create', requireAuth, pageController.getCreateWorkPage);
router.get('/works/:id', pageController.getWorkPage);
router.post('/works/create', requireAuth, workUpload.array('workImages', MAX_WORK_IMAGES), csrfProtect, workController.createWork);
```

Маршруты, связанные с созданием/изменением данных, защищаются `requireAuth` и `csrfProtect`. Это предотвращает несанкционированную публикацию работ и CSRF-атаки через внешние формы.

#### 4.1.4 Лента работ: как формируются карточки

Страница `/lenta` — центральный рабочий экран платформы. Здесь пользователь просматривает проекты, фильтрует выдачу, ставит лайки и переходит к деталям.

На стороне backend контроллер выполняет несколько шагов:

1. Определяет `currentUserId` из сессии.
2. Гарантирует существование таблицы `work_likes` (если отсутствует, создаёт).
3. Выполняет SQL с JOIN/подзапросами и собирает для каждой работы:
   - автора,
   - массив изображений,
   - список категорий,
   - список ID категорий (для фильтра на фронте),
   - флаг `is_liked` для текущего пользователя.
4. Отдельно получает корневые категории и подкатегории.
5. Передаёт всё в шаблон `lenta_new.ejs`.

**Листинг 3 – Состав данных карточки в контроллере ленты (идея запроса)**

```sql
SELECT 
  w.*, u.id as user_id, u.first_name, u.last_name, u.avatar,
  ... as images,
  ... as categories,
  ... as category_ids,
  ... as is_liked
FROM works w
JOIN users u ON w.user_id = u.id
WHERE w.status = 'active'
ORDER BY w.created_at DESC
```

На стороне шаблона каждая запись рендерится как карточка `.post` с атрибутами `data-work-id` и `data-work-category-ids`. Эти data-атрибуты затем используются JavaScript-логикой:
- для открытия модального окна жалобы;
- для фильтрации по категориям без перезагрузки страницы;
- для обработки лайка и визуального состояния кнопки.

#### 4.1.5 Галерея изображений в карточке и на странице работы

Для карточки ленты и для страницы `/works/:id` используется одинаковая UX-идея: горизонтальный трек изображений и стрелки `prev/next`.

Алгоритм переключения:
1. В JS берётся контейнер `track` и список `slides`.
2. Хранится текущий индекс `index`.
3. При нажатии стрелки индекс увеличивается/уменьшается в допустимых пределах.
4. Вызывается `render()`, который делает `translateX(-index * 100%)`.
5. Кнопки отключаются на границах (первый/последний слайд).

**Листинг 4 – Логика render() для галереи**

```js
const render = () => {
  track.style.transform = `translateX(-${index * 100}%)`;
  prev.disabled = slides.length <= 1 || index === 0;
  next.disabled = slides.length <= 1 || index === slides.length - 1;
};
```

Если изображение недоступно, в `img` используется fallback (`/img/work_img.svg`) через `onerror`, что гарантирует корректный внешний вид даже при «битых» ссылках.

#### 4.1.6 Фильтрация по категориям и модерационные действия

В правой части ленты размещён виджет фильтра. Категории и подкатегории приходят с сервера, а фильтрация происходит на клиенте по `data-work-category-ids`.

Пользовательский сценарий:
1. Пользователь выбирает несколько категорий.
2. Скрипт обновляет счётчик выбранных пунктов.
3. Скрипт скрывает/показывает карточки, проверяя пересечение выбранных ID с ID работы.
4. При «Сбросить всё» фильтр очищается и показываются все карточки.

Дополнительно в ленте присутствует жалоба на работу (модальное окно). Форма включает список типовых причин, скрытое поле причины и CSRF-токен.

#### 4.1.8 Профиль пользователя `/profile` и `/profile/:id`

Контроллер профиля агрегирует данные из нескольких таблиц:
- `users`, `roles` — персональные данные и роль;
- `user_reviews` — средняя оценка и количество отзывов;
- `works`, `work_images`, `work_categories`, `categories` — портфолио;
- `subscriptions` — число подписчиков и состояние подписки текущего пользователя.

Особенность реализации: если пользователь открыл **собственный** профиль, дополнительно подгружаются `pending`-работы. В публичном просмотре эти работы не отображаются.

Таким образом, профиль одновременно работает как:
1. публичная витрина исполнителя;
2. личный кабинет автора для контроля статуса публикаций.

#### 4.1.9 Создание работы и загрузка изображений

Страница `/works/create` доступна только авторизованным пользователям. Файлы загружаются через `multer` с дисковым хранилищем в `public/uploads`.

Ограничения загрузки:
- до 10 файлов на работу;
- размер файла до 10 МБ;
- принимаются только MIME-типы `image/*`.

**Листинг 5 – Конфигурация загрузчика изображений**

```js
const workUpload = multer({
  storage: workStorage,
  limits: {
    fileSize: 10 * 1024 * 1024,
    files: MAX_WORK_IMAGES,
  },
  fileFilter: (_req, file, cb) => {
    if (file.mimetype && file.mimetype.startsWith('image/')) {
      return cb(null, true);
    }
    return cb(new Error('Разрешены только изображения'));
  },
});
```

При сохранении файла формируется уникальное имя вида `work-<timestamp>-<random>.<ext>`, что снижает риск коллизий имён при параллельных загрузках.

#### 4.1.11 Безопасность и устойчивость

В текущей реализации уже использованы важные базовые меры:
1. сессионная авторизация через `express-session`;
2. cookie-флаги `httpOnly`, `sameSite`, `secure` (для production);
3. CSRF-токены для форм;
4. разграничение доступа через middleware `requireAuth`.

Практический эффект:
- снижается риск подделки межсайтовых запросов;
- ограничивается доступ к защищённым операциям;
- уменьшается вероятность кражи сессии через клиентские скрипты.

---

#### 4.1.12 Вывод по реализации

Реализация сайта Liniti покрывает полный базовый цикл взаимодействия на платформе:
1. пользователь знакомится с сервисом на лендинге;
2. авторизуется и настраивает профиль;
3. публикует работу с медиаконтентом;
4. другие пользователи находят проект в ленте через фильтрацию;
5. переходят в карточку работы и профиль автора;
6. взаимодействуют через лайки, подписки, жалобы и дополнительные разделы.

Таким образом, в репозитории реализована целостная веб-система с разделением на публичную часть, авторизованные пользовательские сценарии и административную зону, что соответствует требованиям к современному многостраничному веб-приложению.
